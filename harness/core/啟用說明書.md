# HARNESS 啟用說明書

## 1. Purpose / 這份資料包是什麼

這份資料包是給新專案啟用 HARNESS 用的最小啟動包。

它讓你之後開新專案時，不需要重新整理整段 Phase 歷史，只要把這個資料包複製到新專案，照步驟啟動即可。

這份資料包不是產品程式碼，不是 Runtime / SDK，不是 automation，也不是 git push 授權。

## 2. Files Read / Source Files Used

本說明書根據以下資料包來源建立：

- `_coordination/workflow_harness_core/workflow_harness_core_v1_0_freeze.md`
- `_coordination/workflow_harness_core/workflow_harness_core_v1_0_export_profile.md`
- `_coordination/workflow_harness_core/workflow_harness_new_project_usage_manual.md`
- `_coordination/workflow_harness_core/writing_tool_project_startup_packet.md`
- `_coordination/workflow_notes/20260517_32_a_lock_writing_tool_route_contract_decision_note.md`
- 使用者指定的新專案路徑：`D:\writing-tool project folder`

本說明書只整理啟用步驟與放置位置。

## 3. Files Not Read

本說明書未讀：

- protected story / world / character / plot / scene / dialogue body text
- `00_protocol/` body text
- `01_world/` body text
- `03_characters/` body text

- `_prototypes/` body text

- frontend / GUI / editor files
- `_exports/`

- `templates/`

- external writing-tool project body content

## 4. Source Limitations

- 本說明書是啟用流程整理，不是產品實作文件。
- 本說明書不驗證新專案是否已完成 adapter。
- 本說明書不讀 protected creative content。
- 本說明書不建立 Project Adapter。
- 本說明書不建立 product code、tasks、templates、frontend、editor、export。
- 本說明書不啟動 Runtime / SDK。
- 本說明書不建立 automation、schema、registry、database。
- 本說明書不批准 git push。
- 新專案仍需要自己的 Project Adapter 與 adoption evidence。

## 7. PowerShell 複製步驟

先設定來源與目標。

如果你把本資料包保留在原 repo 根目錄，來源通常是：

```powershell
$source = "C:\Users\001\game-dialogue-bible\HARNESS啟用資料包"
$target = "D:\writing-tool project folder"
```

建立新專案需要的資料夾：

```powershell
New-Item -ItemType Directory -Force -Path "$target\harness\core"
New-Item -ItemType Directory -Force -Path "$target\harness\route_contracts"
New-Item -ItemType Directory -Force -Path "$target\_coordination\project_adapter"
New-Item -ItemType Directory -Force -Path "$target\_coordination\workflow_notes"
```

複製 HARNESS Core 啟動文件：

```powershell
Copy-Item "$source\workflow_harness_core_v1_0_freeze.md" "$target\harness\core\"
Copy-Item "$source\workflow_harness_core_v1_0_export_profile.md" "$target\harness\core\"
Copy-Item "$source\workflow_harness_new_project_usage_manual.md" "$target\harness\core\"
Copy-Item "$source\writing_tool_project_startup_packet.md" "$target\harness\core\"
```

複製劇本工具路線邊界：

```powershell
Copy-Item "$source\writing_tool_route_contract.md" "$target\harness\route_contracts\"
```

如果新專案還不是 git repo：

```powershell
cd "$target"
git init
```

## 8. 在新專案開 Codex

用 Codex 打開新專案資料夾，例如：

```text
D:\writing-tool project folder
```

然後在新的 Codex 對話貼上以下啟動文字。

```text
身份：Codex-Phase 1 主控室
狀態：A route writing-tool project startup candidate
Report back to：Codex-Phase 1 主控室（本對話）

目的：
使用 HARNESS 啟動新的 writing-tool project。
先只做 HARNESS adoption / Project Adapter planning，不做產品實作。

硬規則：
1. 只使用 Phase 命名。
2. `PUSH`、`已 PUSH`、`PUSH 繼續`、`長跑`、`commit` 都不等於授權 git push。
3. 除非使用者明確說「你可以 push / 授權你 git push」，不得執行 git push。
4. PASS / READY / VERIFIED 都只是 evidence，不是 approval。
5. 設計室 / 實作室回報只是 supporting evidence，不是 authority。
6. 不讀 protected creative content，除非有 exact read gate。
7. 不做 automation / dispatch / snapshot / authority update。
8. 不把 REVIEW 當 LOCKED。
9. 不把 Core 當 Project Adapter。
10. 不修改 HARNESS，除非遇到重大 blocker。
11. 不建立 product code / tasks / templates / frontend / export，直到 Project Adapter 與第一條產品 gate 確認。

啟動後第一步：
1. live check `git status --short -uall`
2. live check `git status -sb`
3. live check `git log -1 --oneline`
4. 讀 `harness/core/workflow_harness_core_v1_0_freeze.md`
5. 讀 `harness/core/workflow_harness_core_v1_0_export_profile.md`
6. 讀 `harness/core/workflow_harness_new_project_usage_manual.md`
7. 讀 `harness/core/writing_tool_project_startup_packet.md`
8. 讀 `harness/route_contracts/writing_tool_route_contract.md`
9. 提出 Project Adapter exact scope proposal
10. 不直接寫檔，先回報 active state

Output Required：
- active controller 是誰
- Report back target 是誰
- git state
- HARNESS route boundary status
- proposed Project Adapter path
- protected-source initial list
- Files Not Read
- source limitations
- stop conditions
- validation commands
- Push Gate rule confirmation
- 現在在幹嘛
- 下一步要做的事
- 後續建議
```

## 9. 新專案第一個工作項目

新專案第一個工作項目應該是：

```text

```

- 直接開始寫產品功能；
- 直接讀劇本資料；
- 直接建立 frontend / editor / export；
- 直接建立 Runtime / SDK；
- 直接把舊 adapter 複製過去；
- 直接把任何文件升級成 LOCKED。

建議第一個 adapter 目標：

```text
_coordination/project_adapter/writing_tool_project_adapter.md
```

## 10. 停止條件

遇到以下情況要停下來確認：

- active controller 不清楚；
- report-back target 不清楚；
- 新專案 Project Adapter 還不存在；
- 要讀 protected creative content；
- 要複製舊 repo 的劇本內容；
- 要直接複製 `game_dialogue_bible_project_adapter.md`；
- 要修改 HARNESS Core；
- 要建立產品程式碼；
- 要建立 tasks / templates / frontend / export；
- 要啟動 Runtime / SDK；
- 要建立 automation / schema / registry / database；
- 要 commit 但修改範圍不清楚；
